Tests:
the program must include test calls of isOrderedfrom a main method. The tests are required to include (at least)
  • an empty array (expect result: true for both directions)
  • a single element array with element value 0 (expect result: true for both directions)
  • an array that is in order upwards (with both negative and positive values, and including zero) (expect result: true when up is true; false when up is false)
  • out of order at the start (expect result: false for both directions)
  • out of order at the end (expect result: false for both directions)

Constraints and requirements

Submission must be one Java source file named Ordered.java containing 2 or more methods. Two required methds are:

/**
 * determine whether an array is in non-decreasing order upwards or downwards
 * @param values an array of ints of any length
 * @return true if the elements of values are in non-decreasing order
 * that is, each element is equal to or greater than its predecessor-
 * as the array index increases, if up is true; or as it decreases, if up is false
 * otherwise return false
 */
public static boolean isOrdered (int values[], boolean up )

/**
 * test the isOrdered method
 * @param args is ignored
 * @return exit  value shall is 1 if any test fails, otherwise exit value is 0
 */
public static void main(String args[])

Coding style and standards

The program is required to be of good standard and good style. The standard will get stronger as you proceed through the course. For this first homework, here are suggested style and standards.

Source code style

Identifiers
identifiers shall be meaningful to the reader, preferably several characters long ( a single letter is usually not a good choice) formed from English words or phrases that indicate the role of the variable, class or method in the solution. Multiple words in an identifier should be indicated by changes of letter case (such as isOrdered, rather than isordered). Abbreviated phrases are expected where the meaning is still clear: the result should at least have mnemonic relationship.
All upper case is preferred for names of CONSTANT values (and the body of the program code should have no “magic numbers”, only named constants.)
Array index variable (such as i in for (int i ; i < values.length; i++)...) can often be eliminated by using the for-each version of the loop: for (int v: values)... (which is also less code to type, is clearer to read becuase it states the intention of the dcode directly, and carries less risk of making an error in covering all elements of the array). If it is necessary to have an index variable because the body of the loop or the following code refers to it explicitly, a single-letter identifier may suit a mathematical-style algorithm; a short word, or a name related to the array identifier, may be better when an algorithm is traversing a data structure.
Comments
every method and every class shall be preceded by usage documentation in javadoc style (see the examples in this homework specification, and examples of programs in Horstmann Big Java. Schildt Java the complete referenceis a bad example, most code fragments and programs are not sufficiently documented).
Inline comments should be used to label sections within the coding, and briefly explain any unusual or particularly smart program steps.
Test cases should be coded or documented (with comments) in order to be self-explanatory to someone reading the testcases, without reading the implementation of the method being tested.
Layout
Source code should be formatted similar to the indented layout in Eclipse (see the recommended Java Code Conventions.

Coding standards

Code length, generality of solution
Algorithms and data structures should be chosen to make the program short, and handle as many cases and values as possible in a suitably small number of separate cases: without being too short and “tricky”.
Using enhanced loops, library methods, and other code-saving techniques is good style.
Detecting errors and exceptional cases
Errors such as the program arguments not meeting the program specification, or method parameters not meeting the requirements that are documented for the method, shall be detected and reported appropriately. How this is handled: whether an exception is thrown, or an error message output, or a warning, is determined by the specifications given for the program or method. In cases where the specifications are not complete, handling exceptions is determined by the good judgment of the programmer, within the overall specified behavioural constraints and requirements.
Clarity and elegance
Good program code communicates to the human reader, as well as specifying actions to a computer. Like clear writing in a natural language, readable programs need to have clear identifiers and clear operations. They need the programmer to use wordy forms to spell out conditions and choices rather than relying on obscure properties of arithmetic operators; benefit from over-bracketing, and clearly laying out, boolean expressions; and especially from using and creating good abstractions. The tools in IDEs that allow quick and safe renaming of parts of the program are very useful.
Performance and efficiency
Good runtime performance and space perfomance are important, but take second place to clear description and abstraction. But algorithmic efficiency is an important design property: O(n) will nearly always be preferred to O(n2). But saving a single method call, even in an inner loop executed thousands of times, is not worthwhile (the compiler may provide this optimisation: the programmer should not obscure the clarity of the program coding to make such small gains, in ordinary programs. Special purpose high performance work is different).
